# MindClaw

[English](./README.md) | [简体中文](./README.zh-CN.md)

**在强大的运行时核心之上构建自治智能体团队。**

MindClaw 是一个面向自治编码团队的开源多智能体编排外壳。
它希望把通常被拆开的两部分结合起来：

- 强大的单智能体执行能力
- 显式的团队级协作与编排能力

MindClaw 的目标，是让你能够用一种更实际的方式构建 leader-worker 智能体系统：它们可以拆分工作、隔离执行环境、交换消息、跟踪任务状态，并在底层接入不同的运行时引擎。

> 如果你觉得这个方向有价值，欢迎给项目点一个 star，关注后续开发进展。

## 为什么是 MindClaw

很多智能体系统都过于偏向某一个方向：

- 有些单智能体执行能力很强，但团队协作能力很弱
- 有些擅长做流程编排，但底层运行时智能不够强
- 有些内部功能很多，但几乎不暴露显式的团队状态

MindClaw 想解决的正是这个中间地带。

它的目标不是简单地“运行更多智能体”。
它真正想做的是，让多智能体协作变得更可理解、更可组合，也更适合开源项目长期演进。

## MindClaw 想实现什么

通过 MindClaw，你最终应该能够：

- 创建 leader + worker 的智能体团队
- 把工作拆分成任务图和执行阶段
- 让每个 worker 在独立工作区中运行
- 通过结构化状态在智能体之间路由消息
- 在统一的 shell 或面板中观察执行进展
- 在同一套编排模型下接入不同的运行时引擎

## 核心设计思想

### 1. 团队状态必须是显式的

MindClaw 将团队成员、任务、消息、工作区和生命周期都视为一等编排状态。

### 2. 运行时执行必须是可插拔的

MindClaw 不应该被绑定到某一个单独的智能体运行时。它应该通过运行时适配层，位于不同执行引擎之上。

### 3. 隔离能力应该内建

Worker 应该能够在各自独立的工作区中运行，这样并行执行会更容易理解，也更安全。

### 4. 开源可读性很重要

这个仓库正从根目录开始按独立项目重新组织，这样贡献者可以更快理解系统边界、路线图和实现方向。

## 高层架构

```text
User
  -> MindClaw CLI / Shell
    -> Orchestrator
      -> Team / Tasks / Messaging / Workspaces
        -> Runtime Adapter
          -> Agent Runtime
            -> Tools / Memory / Sessions / Models
```

从高层看：

- **MindClaw Shell** 负责编排与团队协作
- **Runtime Adapter** 负责把 worker 意图翻译成具体运行时启动方式
- **Agent Runtime** 负责单智能体执行、记忆、工具和会话

更多细节请见 `docs/architecture.md`。

## 示例使用场景

### 自治软件工程团队

一个 leader 智能体负责规划工作，多个 worker 在隔离的代码工作区中分别实现任务，外层 shell 统一跟踪进度与归属。

### 研究型智能体群

不同 worker 分别探索不同假设、工具链或实验分支，而 shell 负责保持整体团队状态一致。

### 长时间运行的智能体工作流

MindClaw 适合那些需要长期保持会话、任务状态和执行边界可理解的工作流。

### 可复用的团队模板

长期目标是支持常见协作模式的可复用模板，例如工程团队、研究团队和垂直领域 swarm。

## 项目状态

MindClaw 当前处于 **早期开发阶段**。

当前仓库主要聚焦于：

- 建立公开项目结构
- 定义系统架构边界
- 搭建最小 CLI 与 Python 包骨架
- 准备核心编排模型

这意味着仓库已经具备开源开发的基础形态，但更深层的编排能力与运行时集成仍在持续建设中。

## 快速开始

MindClaw 目前仍在持续搭建中，但仓库已经可以安装和运行基础命令。

```bash
git clone https://github.com/wanlixing-dream/MindClaw.git
cd MindClaw
pip install -e .
mindclaw --help
mindclaw doctor
```
